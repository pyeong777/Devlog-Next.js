## 오래된 블로그를 다시 열어봤다

예전에 만든 블로그 코드를 다시 열어보니 손볼 곳이 꽤 많았다. Next.js 버전도 오래됐고, 글을 올리는 방식도 지금 기준으로는 불편했다. 글 하나를 올리려면 `posts.json`에 메타데이터를 추가하고, 같은 slug로 markdown 파일을 만들고, 이미지 파일까지 따로 맞춰야 했다.

작동은 했지만 실수하기 쉬운 구조였다. 블로그는 글을 쓰려고 만든 건데, 정작 글을 올릴 때마다 파일 여러 개를 맞춰야 하는 건 오래 유지하기 어렵다.

이번에는 배포 방식 자체를 크게 바꾸기보다, 앞으로 글을 편하게 올릴 수 있게 만드는 쪽에 집중했다.

## 패키지 매니저부터 정리했다

먼저 패키지 매니저를 npm으로 통일했다. 기존 저장소에는 `package-lock.json`, `yarn.lock`, `.yarn` 캐시가 같이 들어 있었다. 로컬에서는 어떻게든 설치가 되더라도, 시간이 지나면 어떤 lock 파일을 기준으로 봐야 하는지 애매해진다.

그래서 Yarn 관련 파일을 제거하고 npm 기준으로 정리했다. 저장소에 올라가 있던 Yarn cache도 같이 제거했다. 의존성은 설치해서 해결할 문제이지, 블로그 저장소가 패키지 캐시를 직접 들고 있을 필요는 없다고 판단했다.

## Next.js도 최신 버전으로 올렸다

Next.js는 13대 버전에서 16.2.6으로 올렸다. React도 19로 같이 올렸다. 단순히 버전만 바꾸면 끝나는 작업은 아니었다.

Next 16에서는 Turbopack이 기본 빌드 경로가 되면서 예전 webpack 설정이 바로 문제가 됐다. 예전에 넣어둔 SVG용 webpack 설정은 실제로 사용하지 않는 설정이라 제거했다. 설정 파일도 `next.config.mjs`로 정리했다.

또 App Router 쪽 타입도 맞췄다. 동적 라우트의 `params` 타입, metadata 설정, canonical URL, Open Graph 이미지 설정도 같이 정리했다. 빌드 때 나오던 `metadataBase` 경고도 없앴다.

## 글 데이터는 frontmatter로 옮겼다

가장 크게 바꾼 부분은 글 데이터 구조다.

기존에는 글 메타데이터가 `data/posts.json`에 있고, 본문은 `data/posts/*.md`에 있었다. 이러면 글 하나가 두 군데로 나뉜다. 제목이나 slug를 바꿀 때 실수하기 쉽고, 이미지 파일 이름도 따로 맞춰야 했다.

이제는 markdown 파일 상단에 frontmatter를 넣는 방식으로 바꿨다.

```md
---
title: "글 제목"
description: "글 설명"
date: "2026-05-21"
category: "nextjs"
path: "post-slug"
featured: true
image: "/images/ogImage.png"
---
```

이렇게 바꾸면 글 하나가 파일 하나로 닫힌다. 목록 페이지, 상세 페이지, sitemap 모두 markdown의 frontmatter를 읽어서 만든다.

## 글 작성 스크립트를 추가했다

새 글을 만들 때는 이제 아래 명령어를 쓰면 된다.

```bash
npm run new-post
```

제목, 설명, 카테고리, 날짜 정도만 입력하면 markdown 파일을 만들어준다. 기본 커버 이미지는 블로그의 OG 이미지를 사용하도록 했다. 매번 이미지를 준비하지 않아도 글을 올릴 수 있게 하기 위해서다.

물론 특정 글에 맞는 커버가 필요하면 frontmatter의 `image` 값만 바꾸면 된다. 당장 외부 이미지 API를 붙이기보다는, 기본 이미지를 안정적으로 쓰고 필요할 때만 덮어쓰는 방식이 더 낫다고 봤다.

## 빌드 전에 글도 검증한다

글이 많아질수록 사소한 실수도 배포 실패로 이어질 수 있다. 그래서 글 검증 스크립트도 추가했다.

```bash
npm run validate-posts
```

이 스크립트는 frontmatter 필드가 빠졌는지, 날짜 형식이 맞는지, slug가 파일 이름과 같은지, 이미지 경로가 실제로 존재하는지 확인한다.

최종 확인은 아래 명령어 하나로 끝난다.

```bash
npm run check
```

이 명령어는 글 검증, lint, production build를 순서대로 실행한다.

## CI도 붙였다

GitHub Actions도 추가했다. 이제 main 브랜치에 push하거나 PR을 만들면 GitHub에서 `npm ci`, 글 검증, lint, build를 돌린다.

Vercel 배포가 있더라도, 배포 전에 GitHub에서 기본적인 문제를 먼저 잡는 편이 좋다. 특히 블로그처럼 자주 글을 추가하는 프로젝트는 빌드가 깨지는 커밋을 빨리 알아차리는 게 중요하다.

## 남은 작업

이번 정리는 블로그를 다시 쓰기 좋게 만드는 첫 단계에 가깝다. 아직 오래된 글 본문의 인코딩 문제나 About 페이지 문구 정리는 남아 있다. 디자인도 지금 기준으로 보면 더 다듬을 수 있다.

그래도 큰 흐름은 정리됐다. 이제 글 하나를 올리기 위해 여러 파일을 손으로 맞추는 구조는 아니다. 글 작성, 검증, 빌드, 배포까지 이어지는 기본 파이프라인이 생겼다.

블로그를 다시 운영하려면 이 정도 기반은 먼저 있어야 한다고 생각한다. 이제부터는 코드를 고치는 일보다 글을 쓰는 일이 더 쉬워져야 한다.
